# Player Retention and Churn SQL Analysis

## Project Overview

I built this as a SQL-only portfolio project to demonstrate the kind of analysis I would do as a Junior Data Analyst. The project uses a simulated iGaming-style dataset and focuses on player retention, churn risk, cohort behavior, bonus abuse risk, A/B test performance, and data quality validation.

The goal was to go beyond simple SELECT statements and show that I can structure a real analysis using joins, CTEs, aggregations, CASE logic, date filtering, and validation checks.

This project is written from the point of view of an analyst answering business questions directly from raw data.

## Business Questions

For this project, I focused on answering these questions:

1. Which player segments have the highest churn risk?
2. Which high-value players are at risk of leaving?
3. How do registration cohorts behave across Q1 2026?
4. Which campaigns may be creating bonus abuse risk?
5. Did the promotional A/B test improve deposit conversion and revenue?
6. Are there any data quality issues that need to be fixed before trusting the KPIs?
7. What monthly KPI summary would I provide to a business stakeholder?

## Tools and Skills Used

This project demonstrates:

## Dataset

The data is fully simulated and fictional. It represents a small iGaming-style business with players, gameplay sessions, transactions, bonuses, campaigns, and A/B test assignments.

| Dataset | Description |
|---|---|
| `players.csv` | Player profile and acquisition data |
| `sessions.csv` | Gameplay activity, stakes, winnings, and session behavior |
| `transactions.csv` | Deposits, withdrawals, payment methods, and intentional data-quality issues |
| `bonuses.csv` | Bonus awards and redemption status |
| `campaigns.csv` | Campaign names, types, and active dates |
| `ab_test_assignments.csv` | Control vs. Variant test data for deposit conversion and 14-day revenue |

## KPIs and Logic Used

| KPI / Logic | Definition Used |
|---|---|
| GGR | Stakes - Winnings |
| NGR | GGR - Redeemed Bonus Cost |
| Deposit Volume | Sum of valid deposit transactions |
| Withdrawal Volume | Sum of valid withdrawal transactions |
| Churn Risk | No activity for 30+ days as of 2026-04-01 |
| At Risk | No activity for 14-29 days as of 2026-04-01 |
| Active | Activity within the last 14 days |
| Player LTV Proxy | NGR generated by a player during the Q1 observation window |
| Bonus Abuse Flag | High bonus usage relative to deposits, negative NGR, and meaningful bonus amount |
| A/B Test Lift | Variant conversion rate compared against Control conversion rate |

## Key Findings I Would Present

Because this project is designed as a SQL portfolio project, the focus is on the logic and analysis workflow. The simulated dataset is structured to support the following business-style findings:

- Churn risk is not evenly distributed across player segments.
- Some high-value players have gone inactive and should be prioritized for retention campaigns.
- Bonus-heavy campaigns can look successful on conversion while still creating low or negative net revenue.
- The Variant group in the A/B test can be compared against Control by conversion rate, retention, and average 14-day revenue.
- Data quality checks are required before reporting KPIs because the dataset includes duplicate transaction IDs, missing player IDs, negative values, and out-of-range dates.

## How I Would Explain This Project in an Interview

I built this project to show that I can use SQL to answer business questions, not just pull raw data.

The analysis starts by creating a player-level KPI table from multiple sources: player profiles, sessions, transactions, and bonuses. From there, I use SQL to classify players as Active, At Risk, Churn Risk, or No Q1 Activity based on their last activity date.

I also analyze registration cohorts, identify high-value churn-risk players, check bonus abuse risk by campaign, and summarize A/B test performance. The final part of the project focuses on data validation because I wanted to show that I understand the importance of checking data before trusting a report.

If I were walking through this in an interview, I would explain how each query connects to a business question and how the validation checks protect the accuracy of the final analysis.
